# VibeCoding 工作流

本教程详细介绍 VibeCoding 的 7 个核心原则，帮助开发者从传统开发模式高效转向 AI 辅助开发模式。

## 引用说明

本教程的整体框架参考自：
- https://www.vibevibe.cn/Advanced/02-ai-tuning-guide/02-vibecoding-workflow.html

## 1. 角色转变：从"写代码"到"编排 AI"

| 传统开发者    | Agent Native 编排者 |
| -------- | ---------------- |
| 手写每一行代码  | 描述需求，让 AI 实现     |
| 逐个调试 Bug | 反馈问题，让 AI 修复     |
| 关注语法细节   | 关注产品方向           |
| 是工匠      | 是指挥官             |

核心思维：代码是实现细节，产品才是目的。开发者负责"判断"，AI 负责"预测"。

## 2. 标准五步工作流（灵活应用）

```
探索 → 规划 → 执行 → 验证 → 提交
```

| 步骤 | 目的           | 关键动作                    |
| -- | ------------ | ----------------------- |
| 探索 | 了解现状，避免重复造轮子 | 让 AI 扫描项目结构、技术栈、现有功能    |
| 规划 | 先想清楚再动手，减少返工 | 明确需要创建/修改的文件、数据库变更、实现顺序 |
| 执行 | 按计划实现功能      | AI 自动拆分任务、并行处理独立部分      |
| 验证 | 确保功能正常       | 功能测试、类型检查、关键逻辑审查        |
| 提交 | 建立回滚点        | 自动 git commit，生成简洁提交信息  |

灵活原则：

- 复杂功能/不熟悉项目 → 严格遵循五步
- 简单修改/熟悉项目 → 可跳过探索直接执行
- 微小改动 → 最小化流程

## 3. 高质量提示词设计

### 提示词对比表

| 场景类型   | 欠佳提示词       | 推荐提示词                                                                                |
| ------ | ----------- | ------------------------------------------------------------------------------------ |
| 功能开发   | "帮我写一个登录功能" | "实现用户登录：用户名+密码，Next.js 16 App Router，Drizzle ORM + PostgreSQL，包含表单验证和错误处理"           |
| Bug 修复 | "修复这个 Bug"  | "修复 Bug：文件 app/login/page.tsx，问题：登录后未跳转，预期：跳转到 /dashboard，当前代码：\[片段]"                |
| 代码重构   | "优化一下代码"    | "重构 components/Table.tsx：现有问题是渲染性能差，目标是使用 React.memo 和虚拟滚动提升性能，保持现有 API 不变"          |
| 代码审查   | "看看这段代码"    | "审查 lib/api.ts：重点关注错误处理是否完善、是否存在安全隐患、命名是否清晰"                                         |
| 文档编写   | "写点文档"      | "为 utils/helpers.ts 编写 JSDoc：每个函数需要描述用途、参数类型、返回值类型，包含 1-2 个使用示例"                     |
| 测试编写   | "加个测试"      | "为 components/Form.tsx 编写单元测试：使用 Jest + React Testing Library，覆盖正常提交、必填项校验、网络错误三种场景" |
| 架构设计   | "设计一个系统"    | "设计用户管理系统架构：需要实现的功能包括用户 CRUD、权限控制、登录日志，技术栈：Node.js + Express + MySQL"                |
| 上下文提供  | 不提供上下文      | "项目使用 TypeScript 5 + React 18 + Tailwind CSS，已有 User 表和 AuthContext 可直接复用"           |
| 不确定场景  | 强行要求给出答案    | "如果不确定，请明确说'我不确定'，需要补充哪些信息请告知"                                                       |

核心原则：

- 不要让 AI 猜 → 提供明确上下文
- 不要含糊其辞 → 给出具体要求
- 不要强行编造 → 给 AI"不确定"的出口

## 4. 权限模式选择（Claude Code 特有）

| 模式           | 特点        | 适用场景                       |
| ------------ | --------- | -------------------------- |
| Default（推荐）  | 读自动，写询问   | 日常开发，平衡效率与安全               |
| Plan         | 只读不写      | 新功能涉及 3 个以上文件、大规模重构、不熟悉代码库 |
| Accept Edits | 写需确认，其他自动 | 频繁测试/运行命令                  |

快捷键：Shift+Tab 循环切换

Plan 模式深度价值：

- AI 只能使用只读工具，确保规划阶段专注分析
- 自动深度探索代码库，理解架构和依赖
- 输出结构化方案，等审批后再执行

## 5. 多 Agent 并行协作

### 自动并行：使用关键词触发

```
"同时做这三件事：
1. 编写后端 API（用户认证）
2. 编写前端 UI（登录表单）
3. 编写数据库 schema（User 表）"
```

### Agent 团队模式（复杂任务）

```
"从以下角度分析这个 API 设计：
1. 安全性：注入风险、认证漏洞
2. 性能：N+1 查询、缓存策略
3. 可维护性：命名规范、职责划分"
```

自定义 Agent：创建专门助手（如 security-reviewer、code-reviewer），保持主对话简洁。

## 6. 信任但验证机制

不要逐行检查 AI 代码，而是：

- 功能测试：运行程序验证功能
- 类型检查：运行 tsc 等工具检查类型错误
- 代码审查：只看关键逻辑，不看实现细节

检查点功能（快速回滚）：

- 按 Esc+Esc 或 /rewind 回退
- 局限：只回滚 AI 写入的文件，终端命令产生的变更（如 npm install）无法回滚
- 建议：小改动用检查点，大改动先 git commit

更多 Claude Code 命令，请参考 [Claude Code 基本斜杠命令](../extended%20docs/2.a%20Claude%20Code基本斜杠命令.md)。

## 7. 安全与风险控制

### 敏感信息保护

- 不要把 API 密钥、密码、生产数据库连接发送给 AI
- 使用 .env 文件，加入 .gitignore

### 需要人工审查的场景

| 场景      | 原因     | 做法                    |
| ------- | ------ | --------------------- |
| 生产环境部署  | 影响所有用户 | AI 生成方案，人工审核后执行       |
| 数据库结构变更 | 难以回滚   | 先 review schema，再执行迁移 |
| 安全相关代码  | 漏洞后果严重 | 代码审查必不可少              |
| 支付/财务逻辑 | 涉及资金安全 | 测试 + 双人验证             |

## 快速自检清单

发送提示词前检查：

- 任务描述清晰具体
- 提供了必要的上下文（文件路径、现有资源）
- 明确了输出格式和约束条件
- 给了 AI"不确定"的出口
- 避免了冗长的角色设定
- 没有包含敏感信息

## 核心总结

好的 VibeCoding 工作流 = 清晰的意图表达 + 结构化的执行流程 + 适度的信任与验证。

开发者是导演，AI 是演员——开发者把控方向，AI 负责实现。
