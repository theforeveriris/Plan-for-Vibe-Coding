# 构建自己的 Skills

本文档详细介绍如何从零创建和管理自己的 Skill。

## Skill 的核心结构

一个完整的 Skill 通常包含以下几个模块：

```
Skill/
├── README.md              # 技能说明和元数据
├── system.md              # 系统级指令（核心人格和行为准则）
├── rules/                 # 具体规则集
│   ├── coding-style.md    # 代码风格规范
│   ├── tech-stack.md      # 技术栈定义
│   └── constraints.md     # 限制和边界条件
├── knowledge/             # 领域知识库
│   ├── project-structure.md
│   ├── api-reference.md
│   └── patterns/          # 常用代码模式
└── templates/             # 可复用模板
    ├── component.tsx
    └── api-route.ts
```

## 实战：从零创建一个 React + TypeScript Skill

### 第一步：定义系统级指令 (system.md)

这是 Skill 的"灵魂"，定义 AI 的基本身份和行为模式：

```markdown
# Role
你是一位资深全栈工程师，专精于 React 18 + TypeScript 5.x 生态。
你的代码风格以"显式优于隐式"为核心原则。

# Core Principles
1. **类型安全优先**: 绝不使用 `any`，必须正确定义接口
2. **函数式编程**: 优先使用纯函数，副作用隔离在自定义 Hooks 中
3. **性能意识**: 默认使用 React.memo、useMemo、useCallback 进行优化
4. **可访问性**: 所有组件必须考虑 a11y（键盘导航、ARIA 标签）

# Response Format
- 代码块必须使用完整的文件路径作为标题
- 每个函数必须附带 JSDoc 注释
- 修改代码时，先说明"为什么"，再展示"怎么做"
- 如果涉及多个文件，提供文件依赖关系图

# Prohibitions
- 禁止在组件内部直接调用 API（必须通过自定义 Hook）
- 禁止使用内联样式（必须使用 CSS Modules 或 Tailwind）
- 禁止忽略 TypeScript 错误（`@ts-ignore` 必须附带详细原因）
```

### 第二步：建立技术栈规范 (rules/tech-stack.md)

```markdown
# Tech Stack Definition

## 核心框架
- React 18.3+ (Strict Mode enabled)
- TypeScript 5.4+
- Vite 5.x (构建工具)

## 状态管理
- 服务端状态: TanStack Query (React Query) v5
- 客户端状态: Zustand (轻量级，避免 Redux 样板代码)
- 表单状态: React Hook Form + Zod 校验

## 样式方案
- Tailwind CSS 3.4+
- CSS Modules（仅用于复杂动画）
- 设计系统: Radix UI Primitives（无样式组件库）

## 测试策略
- 单元测试: Vitest + React Testing Library
- E2E: Playwright
- 覆盖率门槛: 80% 行覆盖率

## 代码质量
- ESLint: typescript-eslint + react-hooks 规则集
- Prettier: 集成 ESLint，避免冲突
- Husky: pre-commit 钩子运行 lint-staged
```

### 第三步：定义代码模式 (knowledge/patterns/)

**自定义 Hook 模式 (patterns/hooks.md):**

```typescript
// hooks/useUser.ts
import { useQuery } from '@tanstack/react-query';
import { z } from 'zod';

// 1. 定义 Zod Schema（运行时校验）
const UserSchema = z.object({
  id: z.string().uuid(),
  email: z.string().email(),
  name: z.string().min(1),
  role: z.enum(['admin', 'user', 'guest']),
});

type User = z.infer<typeof UserSchema>;

// 2. API 函数独立，便于测试
const fetchUser = async (id: string): Promise<User> => {
  const response = await fetch(`/api/users/${id}`);
  if (!response.ok) throw new Error('Failed to fetch user');
  const data = await response.json();
  return UserSchema.parse(data); // 运行时校验
};

// 3. Hook 封装查询逻辑
export const useUser = (id: string) => {
  return useQuery({
    queryKey: ['user', id],
    queryFn: () => fetchUser(id),
    enabled: !!id, // 防止空 ID 请求
    staleTime: 5 * 60 * 1000, // 5 分钟缓存
  });
};
```

**关键规则**

- 每个 Hook 只做一件事（单一职责）
- 必须返回带有明确类型的对象
- 错误处理必须在 Hook 内部完成，不抛给组件
- 依赖数组必须完整（ESLint exhaustive-deps）

### 第四步：创建文件模板 (`templates/`)

**组件模板** (`templates/component.tsx`):

````typescript
import React, { memo } from 'react';
import { cn } from '@/lib/utils'; // 假设的 className 合并工具

interface {{ComponentName}}Props {
  /** 组件的主要标识 */
  id: string;
  /** 额外的 CSS 类名 */
  className?: string;
  /** 点击回调 */
  onAction?: (id: string) => void;
  /** 子元素 */
  children?: React.ReactNode;
}

/**
 * {{ComponentName}} 组件
 *
 * @description
 * [在此处添加组件功能描述]
 *
 * @example
 * ```tsx
 * <{{ComponentName}} id="example" onAction={handleAction}>
 *   Content here
 * </{{ComponentName}}>
 * ```
 */
export const {{ComponentName}} = memo<{{ComponentName}}Props>(({
  id,
  className,
  onAction,
  children,
}) => {
  const handleClick = React.useCallback(() => {
    onAction?.(id);
  }, [onAction, id]);

  return (
    <div
      className={cn('base-styles', className)}
      onClick={handleClick}
      role="button"
      tabIndex={0}
      aria-label={`Action for ${id}`}
    >
      {children}
    </div>
  );
});

{{ComponentName}}.displayName = '{{ComponentName}}';
````

## Skill 的进阶技巧

### 1. 上下文分层管理

不要试图把所有规则写在一个文件里。按作用域分层：

- **Global**: 跨项目通用（如代码风格、Git 规范）
- **Project**: 当前项目特定（如技术栈、架构决策）
- **Task**: 当前任务特定（如"实现登录功能"的临时约束）

### 2. 动态上下文注入

在 Skill 中预留变量，使用时动态替换：

```markdown
## Current Task Context
- 当前分支: {{branch}}
- 目标功能: {{feature}}
- 相关文件: {{related_files}}
- 禁止修改: {{protected_files}}
```

使用时通过脚本或手动填充：

```bash
# 假设的 CLI 工具
vibe-task --feature="用户认证" --branch="feat/auth"
```

### 3. 版本化与迭代

Skill 也需要版本管理：

```markdown
# Skill Changelog

## v1.2.0 (2024-04-20)
- 新增: Next.js App Router 支持
- 修改: 状态管理从 Redux 迁移到 Zustand
- 废弃: Pages Router 的旧数据获取模式

## v1.1.0 (2024-03-15)
- 新增: React Server Components 规范
- 修复: TypeScript 严格模式下的类型推断问题
```

### 4. 与外部工具集成

```markdown

- [ ] 检查 `src/components` 是否已存在类似组件

## Post-Generation Checklist
- [ ] 添加 Storybook 故事文件
- [ ] 更新 `CHANGELOG.md`
- [ ] 运行测试：`npm test -- {{componentName}}`
```

## 快速启动模板

如果你现在就想开始，这是最小可用的 Skill 结构：

```markdown
# My Vibe Coding Skill (最小版本)

## Identity
你是一位注重代码质量的 React 开发专家。

## Tech Stack
React 18, TypeScript, Tailwind CSS, Vite

## Rules
1. 使用函数组件 + Hooks
2. 所有 Props 必须定义接口
3. 禁止使用 any
4. 组件导出必须使用 named export
5. 样式使用 Tailwind，禁止内联 style

## File Patterns
组件: src/components/{Name}.tsx
类型: src/types/{name}.ts
工具: src/utils/{name}.ts

## Response Rules
- 代码块标明文件路径
- 先给思路，再给代码
- 主动指出潜在问题
```

## 关键建议

- **从痛点出发**：不要试图一次性写出完美的 Skill。先记录你在 AI 编程中反复纠正 AI 的问题，把这些纠正写成规则。
- **保持简洁**：规则过多会导致 AI"失忆"或混淆。核心规则不超过 10 条，详细规范放在引用文档中。
- **持续迭代**：每完成一个项目，回顾哪些规则生效了，哪些被 AI 忽略了，调整优先级和表述方式。
- **区分"必须"和"建议"**：
  - 必须（MUST）：使用强制语气，AI 必须遵守
  - 建议（SHOULD）：使用建议语气，AI 参考执行
  - 可以（MAY）：提供额外选项
